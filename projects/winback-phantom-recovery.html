Reactivation Pipeline + Match Recovery

A reactivation pipeline plus data-quality recovery: detected an enrichment matching failure, restored affected firmographics, wrote a blameless postmortem, and codified the prevention pattern.

Role: Sole designer, owner of the postmortem and recovery Stack: BigQuery, Python, Cursor workspace rules, shared SQL patterns Period: 2026 Q1-Q2

Problem: The Pipeline

The business had dormant customer and closed-lost cohorts with no systematic reactivation motion. We needed a pipeline that produced a ranked, scrubbed, contact-grain export for sales follow-up.

Problem: The Bug

The first run exposed a serious data-quality problem: a subset of accounts had firmographic fields from an unrelated company. The customer-to-account mapping was intact, but the enrichment columns were wrong.

Root Cause

The root cause was a common SQL trap: upstream enrichment data represented missing URLs inconsistently, and the matching logic treated one form of "blank" as valid. That allowed unrelated firmographics to pass through a deduplication step and appear on affected customer records.

I removed the faulty matching path, added explicit guardrails for blank keys, and moved the corrected pattern into reusable SQL so the same class of bug would be caught before future merges.

What I Built: Recovery

  • Diagnostic query: identified affected rows and confirmed the core customer-to-account mapping was trustworthy while enrichment columns were corrupted.
  • Recovery script: computed a stronger alternate matching key for each affected account, rejoined vendor firmographics, and backfilled recovered industry, revenue, and employee fields where confidence was sufficient.
  • Code hardening: removed the faulty matching path and added explicit guardrails for blank enrichment keys.
  • Workspace rule: always-on guardrail documenting the failure pattern with safe and unsafe examples plus a diagnostic signature for future regressions.
  • Shared SQL patterns: reusable, trust-tiered vendor joins and a monthly bridge data-quality audit.
  • Postmortem: blameless write-up covering what went wrong, what was still trustworthy, what was recovered, and how future recurrence would be caught.

Impact

  • Recovered 82% of affected firmographics without manual account research.
  • The workspace rule caught the same blank-key failure pattern in later query reviews before merge.
  • The shared SQL patterns became the single source of truth for trust-tiered vendor joins across related workflows.
  • Postmortem and recovery script became reusable templates for future vendor-data regressions.

Lesson

Empty-string handling is the new null-handling. Every SQL pattern that joins on user-typed or imported URL data needs to check both. A good postmortem is also a leadership signal: the team trusts you with bigger bugs after you handle one well.
